## Быстрый чеклист

### Обязательно (Блокирующие)
- [ ] Доменный слой не имеет зависимостей от инфраструктуры
- [ ] Агрегаты обеспечивают свои инварианты
- [ ] Value Objects неизменяемы (immutable)
- [ ] Интерфейсы репозиториев в domain, реализация в infrastructure
- [ ] Нет анемичных доменных моделей (поведение в сущностях)

### Желательно (Предупреждения)
- [ ] Единый язык (Ubiquitous Language) используется в коде
- [ ] Агрегаты маленькие и сфокусированные
- [ ] Доменные события для коммуникации между агрегатами
- [ ] Одна транзакция на один агрегат
- [ ] Агрегаты ссылаются на другие агрегаты только по ID

### Хорошо бы
- [ ] Паттерн Specification для сложных запросов
- [ ] Паттерн Factory для сложного создания
- [ ] Доменные сервисы без состояния (stateless)
- [ ] Четкие границы ограниченных контекстов

## Стратегический дизайн

### Оценка ограниченных контекстов

| Проверка | Проходит | Не проходит |
|----------|----------|-------------|
| Контекст имеет четкие документированные границы | ✓ | ✗ |
| Единый язык определен и используется | ✓ | ✗ |
| Нет утечек моделей между контекстами | ✓ | ✗ |
| Карта контекстов документирована | ✓ | ✗ |
| Паттерны интеграции явно выбраны | ✓ | ✗ |

### Вопросы для проверки

**Четкость границ:**
- Можете объяснить, где начинается и заканчивается этот контекст?
- С какими другими контекстами он взаимодействует?
- Есть ли термины, которые означают разное в разных контекстах?

**Согласованность языка:**
- Совпадают ли имена классов с терминологией эксперта предметной области?
- Понял бы эксперт предметной области этот код?
- Есть ли комментарии-пояснения о преобразовании технических терминов в бизнес-термины?

### Красные флаги 🚩
- Один и тот же класс сущности используется в разных контекстах
- "Общие" пакеты между ограниченными контекстами
- Таблицы БД используются несколькими контекстами
- Общие термины вместо доменных (Entity, Item, Data)

## Тактический дизайн

### Чеклист для Entity

- [ ] Имеет идентификатор (ID)
- [ ] Равенство основано на идентификаторе, а не на атрибутах
- [ ] Содержит поведение, а не только getters/setters
- [ ] Самостоятельно проверяет свои инварианты
- [ ] Инкапсулирует изменения состояния

**Хороший пример:**

```typescript
class Order {
  place(): void {
    if (this._lines.length === 0) {
      throw new EmptyOrderError();
    }
    this._status = OrderStatus.PLACED;
  }
}
```

**Плохой пример:**
```typescript
class Order {
  status: string; // Нет инкапсуляции
}

// Логика вне сущности
function placeOrder(order: Order): void {
  if (order.lines.length === 0) {
    throw new EmptyOrderError();
  }
  order.status = 'placed'; // Прямая мутация
}
```

### Чеклист для Value Object

- [ ] Неизменяемый (immutable)
- [ ] Равенство по всем атрибутам
- [ ] Само-валидирующийся при создании
- [ ] Методы без побочных эффектов (возвращают новые экземпляры)
- [ ] Нет идентификатора


**Хороший пример:**
```typescript
class Money {
  private constructor(
    readonly amount: number,
    readonly currency: string
  ) {}
  
  add(other: Money): Money {
    return new Money(this.amount + other.amount, this.currency);
  }
}
```

**Плохой пример:**
```typescript
class Money {
  amount: number; // ❌ Мутабельный!
  currency: string;
  
  add(other: Money): void {
    this.amount += other.amount; // ❌ Мутация!
  }
}
```

### Чеклист для Repository

- [ ] Интерфейс в доменном слое
- [ ] Реализация в инфраструктурном слое
- [ ] Один репозиторий на корень агрегата
- [ ] Возвращает полностью восстановленные агрегаты
- [ ] Нет утечек логики запросов в домен

**Хороший пример:**
```typescript
// domain/repositories/IOrderRepository.ts
interface IOrderRepository {
  findById(id: OrderId): Promise<Order | null>;
  save(order: Order): Promise<void>;
}

// infrastructure/repositories/PrismaOrderRepository.ts
class PrismaOrderRepository implements IOrderRepository {
  async findById(id: OrderId): Promise<Order | null> {
    // Prisma реализация
  }
}
```

### Чеклист для Domain Event

- [ ] Назван в прошедшем времени (OrderPlaced, а не PlaceOrder)
- [ ] Неизменяемый (immutable)
- [ ] Содержит все данные, необходимые обработчикам
- [ ] Использует доменный язык
- [ ] Имеет timestamp и уникальный ID

## Архитектура

### Проверка зависимостей слоев

```markdown
Разрешенные зависимости:

Presentation → Application → Domain ← Infrastructure
              ↓
           Domain

НЕ разрешены:

Domain → Infrastructure  ✗
Domain → Presentation   ✗
Application → Presentation ✗
```

### Проверка Dependency Injection
- [ ] Доменные сервисы получают интерфейсы, а не реализации
- [ ] Прикладные сервисы получают интерфейсы репозиториев
- [ ] Инфраструктура предоставляет реализации
- [ ] Нет прямого создания инфраструктурных объектов в домене

### Проверка CQRS (если применяется)
- [ ] Команды и запросы четко разделены
- [ ] Обработчики команд возвращают void или только ID
- [ ] Обработчики запросов возвращают DTO, а не доменные объекты
- [ ] Read модели оптимизированы для запросов
- [ ] Write модели оптимизированы для поведения

### Проверка Event Sourcing (если применяется)
- [ ] Все изменения состояния через события
- [ ] События неизменяемые факты
- [ ] Агрегат может быть восстановлен из событий
- [ ] Хранилище событий только для добавления (append-only)
- [ ] Снапшоты для производительности (если нужно)

## Качество кода

### Соглашения по именованию

| Элемент | Соглашение | Пример |
|---------|------------|---------|
| Агрегат | Существительное (ед.ч.) | `Order`, `Customer` |
| Сущность | Существительное (ед.ч.) | `OrderLine`, `Address` |
| Value Object | Существительное (описательное) | `Money`, `EmailAddress` |
| Доменный сервис | Глагол + Существительное | `PricingService`, `InventoryChecker` |
| Репозиторий | Агрегат + Repository | `OrderRepository` |
| Доменное событие | Существительное + Причастие прош. | `OrderPlaced`, `PaymentReceived` |
| Команда | Глагол + Существительное | `PlaceOrder`, `CancelOrder` |
| Запрос | Get + Существительное | `GetOrderDetails`, `FindOrdersByCustomer` |

### Ожидаемое покрытие тестами

| Слой | Покрытие | Фокус |
|------|----------|-------|
| Domain | 90%+ | Бизнес-правила, инварианты |
| Application | 80%+ | Оркестрация use case'ов |
| Infrastructure | 70%+ | Интеграция, маппинг |
| Presentation | 60%+ | Обработка запросов/ответов |

### Проверка документации
- [ ] Глоссарий единого языка существует
- [ ] Границы агрегатов документированы
- [ ] Карта контекстов доступна
- [ ] Ключевые доменные правила документированы в комментариях
- [ ] Записи архитектурных решений для важных выборов

## Частые проблемы

### Проблема: Анемичная доменная модель

**Симптом:** У сущностей только getters/setters, вся логика в сервисах

```typescript
// ❌ ПЛОХО
class Order {
  status: string;
  lines: OrderLine[];
}

class OrderService {
  placeOrder(order: Order): void {
    if (order.lines.length === 0) {
      throw new Error('Order is empty');
    }
    order.status = 'placed';
  }
}
```

**Исправление:** Перенести поведение в сущности

```typescript
// ✅ ХОРОШО
class Order {
  place(): void {
    if (this._lines.length === 0) {
      throw new EmptyOrderError();
    }
    this._status = OrderStatus.PLACED;
  }
}
```

### Проблема: Бог-агрегат (God Aggregate)

**Симптом:** Один агрегат с 10+ сущностями, проблемы с производительностью

**Исправление:** Разделить на меньшие агрегаты с возможной консистентностью


### Проблема: Инфраструктура в Domain

**Симптом:** Доменные классы импортируют базу данных, HTTP или другую инфраструктуру

```typescript
// ❌ ПЛОХО
import { PrismaClient } from '@prisma/client'; // ❌ Утечка!

class Order {
  id: string;
  // ...
}
```

**Исправление:** Домен должен быть чистым

```typescript
// ✅ ХОРОШО - domain/model/Order.ts
class Order {
  id: OrderId;
  // ❌ Нет импортов из Prisma!
}

// infrastructure/persistence/PrismaOrderRepository.ts
// ✅ Только здесь используется Prisma
```


### Проблема: Отсутствие единого языка

**Симптом:** Код использует технические термины, эксперты не могут его прочитать

```typescript
// ❌ ПЛОХО
class OrderDTO {
  processRecord(): void {}
}
```

**Исправление:** Использовать доменные термины

```typescript
// ✅ ХОРОШО
class Order {
  place(): void {}
}
```

### Проблема: Ссылка на агрегат по объекту

**Симптом:** Агрегат хранит ссылку на другой агрегат как объект

```typescript
// ❌ ПЛОХО
class Order {
  customer: Customer; // ❌ Полный объект!
}
```

**Исправление:** Ссылаться только по ID

```typescript
// ✅ ХОРОШО
class Order {
  customerId: CustomerId; // ✅ Только ID!
}
```


### Проблема: Транзакция через несколько агрегатов

**Симптом:** Одна транзакция изменяет несколько агрегатов

```typescript
// ❌ ПЛОХО
function placeOrder(order: Order, inventory: Inventory): void {
  order.place();
  inventory.reserve(order.items); // ❌ Та же транзакция!
  await db.commit();
}
```

**Исправление:** Использовать доменные события для возможной консистентности

```typescript
// ✅ ХОРОШО
function placeOrder(order: Order): void {
  order.place(); // Генерирует OrderPlaced событие
  await db.commit();
}

// Отдельный обработчик
function onOrderPlaced(event: OrderPlaced): void {
  inventory.reserve(event.items);
  await db.commit();
}
```


### Проблема: Неправильные импорты в пакетах

**Симптом:** Domain импортирует из Infrastructure

```typescript
// ❌ ПЛОХО - domain/entities/user/User.ts
import { PrismaClient } from '@prisma/client'; // ❌ НЕЛЬЗЯ!
```

**Исправление:** Domain должен быть чистым

```typescript
// ✅ ХОРОШО - domain/entities/user/User.ts
// ❌ НЕТ импортов из @prisma/client
// ❌ НЕТ импортов из @trpc/server
// ✅ Только чистый TypeScript
```

## 🎯 Итог

### Обязательные проверки при Code Review:

1. **Domain**
    - [ ] Нет импортов из `@prisma/client`
    - [ ] Нет импортов из `@trpc/server`
    - [ ] Нет импортов из `next`
    - [ ] Сущности содержат бизнес-логику
    - [ ] Value Objects неизменяемые

2. **Application**
   - [ ] Use Cases используют Domain интерфейсы
   - [ ] Нет прямой работы с Prisma
   - [ ] Возвращают Result<T>

3. **Infrastructure**
   - [ ] Реализует Domain интерфейсы
   - [ ] Использует Prisma только здесь
   - [ ] Маппит Prisma ↔ Domain

4. **Presentation**
   - [ ] Использует Use Cases из Application
   - [ ] Не содержит бизнес-логики
   - [ ] MobX store на клиенте
